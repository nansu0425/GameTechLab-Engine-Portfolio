# Picking 최적화

> **상태**: 본문 개요 초안입니다. **성과 우선 + 그림·코드 중심 + 1200자 내외**의 짧고 시각적인 문서로 설계합니다. repo 분석 후 시각자료, 코드 스니펫, 정량 수치를 채워 본문으로 발전시킬 예정입니다.

## 구조 개요 (성과 우선, 그림·코드 중심)

| 섹션 | 핵심 메시지 | 시각자료 | 분량 |
|------|------------|---------|------|
| 1. 성과 (한눈에) | 표준 씬 기준 picking 비용의 Before/After 비교를 첫 화면에 노출, 면접관에게 30초 안에 인상 전달 | Before/After 비교 (이미지 또는 데모) | ~100자 |
| 2. 배경 | 주차 과제, 표준 씬, 측정부터의 접근 (출발점·측정 도구 통합) | 표준 씬 캡처 (객체·삼각형 규모 라벨) | ~200자 |
| 3. 시도한 최적화 — 다층 접근 | 5단계 미니 사이클, 각 단계는 그림 + 코드 + 1~2문장 | 단계별 다이어그램 5개 + 코드 스니펫 5개 + 누적 효과 그래프 | ~700자 |
| 4. 선택의 근거와 트레이드오프 | 5단계 핵심 선택의 이유(대안과 비교), 채택의 비용, 한계 — 기술적 판단 능력을 명시적으로 전달 | (선택) 대안 비교 / 트레이드오프 표 | ~200~300자 |

**총 본문 분량**: 약 1200자 (한국어 본문 기준, 그림 캡션·표·코드 제외). 본문 통독 ~3분, 코드 스캔 포함 시 ~5분.

## 1장 — 성과 (한눈에)

- 본문 가장 첫 화면. **Before/After 비교**를 큰 시각자료로 배치.
  - 형식 후보: (a) 같은 표준 씬에서의 Before/After 정적 비교 이미지, (b) 측정 수치 Before/After 한눈 비교 (시각화 또는 표), (c) 두 가지를 결합.
- 한 줄 요약: "표준 씬 기준 picking 비용 X → Y, 단계별로 어떻게 도달했는지 본문에서 풀어갑니다." (수치는 repo 분석 시 확정)

## 3장 — 시도한 최적화 단계 미리보기

각 하위 섹션 = **그림 1개 + 코드 스니펫(5~10줄) + 1~2문장 (관찰 → 적용 → 효과)**. 텍스트는 그림·코드를 보완.

1. **가속 구조 도입** — Scene BVH + Binned SAH
   - 그림: BVH 트리 / SAH 분할 직관 도식
   - 코드: BVH 구축 또는 SAH 분할 핵심 부분
2. **동적 씬 대응** — Dirty Flag Refit
   - 그림: 변경된 노드만 갱신하는 흐름 도식
   - 코드: Dirty Flag 마킹·Refit 로직 핵심
3. **트래버설 가지치기** — Front-to-back + Back-face Culling
   - 그림: 가까운 노드 우선 순회 + 카메라 반대 면 컬링
   - 코드: 트래버설 함수의 정렬·조기 종료·컬링 부분
4. **더 좁은 그라뉼래리티** — 2-Level (Mesh BVH 추가)
   - 그림: Scene BVH + Mesh BVH 2단 구조 도식
   - 코드: 2-Level 트래버설 진입 부분 (Scene → Mesh)
5. **캐시 효율** — SoA 데이터 레이아웃
   - 그림: AoS vs SoA 메모리 레이아웃 비교
   - 코드: SoA 자료구조 정의 또는 트래버설 핫패스

**누적 효과 종합** — 단계별 누적 효과 그래프 (1장 Before/After와 다른 각도로, 단계별 기여도 디테일 노출)

## repo 분석 단계에서 채울 항목

- [ ] Picking Profiler가 측정한 정확한 지표
- [ ] 표준 씬 정보 (객체·삼각형 규모, 캡처 이미지)
- [ ] 측정 하드웨어
- [ ] 단계별 정량 수치 (before / after)
- [ ] Before/After 비교 자료 (1장 Hook용) + 단계별 누적 효과 그래프 (3장 종합용)
- [ ] 단계별 다이어그램 5개의 자료 출처 (직접 그릴지, 코드/스크린샷 활용할지)
- [ ] 단계별 코드 스니펫 5개 (각 5~10줄, 단순화된 실제 코드 발췌 + 출처 commit 링크)

## 시각자료 가이드 (그림·코드 우선)

- **Hook (1장)**: Before/After 비교 이미지 또는 데모. 30초 안에 핵심 인상 전달.
- **표준 씬 캡처 (2장)**: 1장. 규모(객체·삼각형 수) 라벨로 컨텍스트 부여.
- **단계별 다이어그램 (3장)**: 5단계 각 1장. 직관적 그림이 텍스트보다 의미 전달이 강함.
- **단계별 코드 스니펫 (3장)**: 5단계 각 1블록(5~10줄). 단순화된 실제 코드 발췌 + 출처 commit 링크.
- **누적 효과 그래프 (3장 끝)**: 1장 Hook과 다른 각도(예: 단계별 누적 막대 그래프).
- **대안 비교 / 트레이드오프 표 (4장, 선택)**: SAH vs Median, 1-Level vs 2-Level, AoS vs SoA 등 **선택 vs 대안** + 메모리·빌드 등의 비용.

## 톤·기타 가이드

- **톤**: README와 일관된 격식체, 1인칭 "제가". 텍스트는 짧고 그림·코드를 보완.
- **비중**: 3장이 절반 이상. 본문 무게 중심은 텍스트가 아니라 **그림과 코드**.
- **4장의 역할**: 단순 사실 나열이 아니라 5단계 핵심 선택의 **근거(대안과의 비교)**를 정리해 **기술적 판단 능력**을 명시적으로 전달. 3장이 "무엇을 했는가", 4장이 "왜 그렇게 골랐는가"를 분담.
- **코드 스타일**: 단순화된 실제 코드 발췌 (긴 함수의 핵심만). 의사코드는 지양.
